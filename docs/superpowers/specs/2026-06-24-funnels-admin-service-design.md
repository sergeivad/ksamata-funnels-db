# Сервис управления воронками (Funnels Admin) — дизайн

**Дата:** 2026-06-24
**Статус:** одобрено пользователем, Фаза 1
**Репозиторий:** `ksamata-funnels-db` (приложение живёт рядом с БД)

## 1. Зачем

Графический сервис-реестр для управления «Проектными воронками» поверх существующей
SQLite-базы `ksamata_funnels.db`. Заменяет ручное редактирование БД через Python-скрипты
на удобный, красивый веб-интерфейс: смотреть, добавлять, редактировать, дублировать,
активировать/деактивировать воронки и управлять справочниками.

Сервис — **источник правды для атрибуции рекламных расходов**: коды воронок (`f7`, `f15`…)
по UTM-стандарту префиксуются к именам кампаний (`f23_cold_video`), и расход матчится на
воронку по этому коду. Поэтому код `f#` — первоклассное поле интерфейса.

Контекст UTM-стандарта: `../UTM-стандарт/CLAUDE.md` и
`docs/UTM-стандарт-v2.0.md` в той папке.

## 2. Стек и деплой

- **Next.js (App Router) + TypeScript** — фронт и API в одном процессе.
- **Tailwind CSS + lucide-react** — точное повторение существующего дизайн-языка
  (кремовый фон, почти-чёрные акценты `#111`, моно-чипы кодов, мягкие карточки,
  пилюли статусов). Источник стиля: `../UTM-стандарт/preview.png` и HTML-макет
  «Проектные воронки».
- **Drizzle ORM + better-sqlite3** — типизированный доступ к существующей схеме.
- **Деплой:** один Docker-контейнер через **Dokploy** на `ssh_server_ksamata_prod`.
  Файл БД — на **персистентном volume** Dokploy (переживает редеплои). Сборка из Git.
- Python-скрипты (`add_*.py`) остаются как есть — это утилиты миграции данных, трогают
  тот же файл БД, переписывать не нужно.

**Альтернатива (отклонена):** FastAPI + отдельная React-сборка — два процесса/языка,
сложнее деплой.

## 3. Изменения схемы БД (миграция в Фазе 1)

Добавляются две колонки в `funnels` (обе nullable/с дефолтом, обратносовместимо):

| Колонка | Тип | Назначение |
|---|---|---|
| `status` | TEXT DEFAULT 'active' | `active` / `draft` — статус из фронта (Активна/Черновик). Сейчас в БД его нет. |
| `front_code` | TEXT DEFAULT '' | Фронтовый код воронки (`f33`). `f#` ≠ `num` для новых воронок — храним явно. |

Существующим 32 воронкам: `status='active'`, `front_code` = `f{num}` (для f1–f26 совпадает
с фронтом; для f27–f32, где фронт разошёлся, проставляется вручную по факту).

Остальная схема не меняется. Справочные таблицы: `products`, `contractors`, `sources`,
`tags`. Связи: `funnels` → `funnel_tags` (tag_type: reg/time_19/time_15) → `tags`.

## 4. Объём Фазы 1

Редактируемое ядро воронки:
- `front_code` (`f#`), `num`, `status` (active/draft)
- продукт / подрядчик / источник (FK на справочники)
- АВ-оси через теги: Продукт / Подрядчик / Канал / Направление (+ стандартные
  Автоворонка / Этап / Время)
- `variant`, `product_name`, `landing_url`, `start_date`, `block_name`

Действия:
- Список воронок (карточки как на макете): чип `f#`, заголовок, пилюля статуса,
  кнопки активировать/деактивировать, дублировать, удалить.
- Создать воронку (форма ядра + выбор/создание продукта/подрядчика/источника).
- Редактировать воронку.
- Дублировать (копия с новым `num`/`front_code`).
- Удалить (с подтверждением; каскад на `funnel_tags`).
- Управление справочниками: список + добавление продуктов / подрядчиков / источников /
  тегов (чтобы заводить новые продукты вроде **ТКМ** прямо из UI).

**Вне Фазы 1 (Фаза 2+):** редактор `funnel_days` (10 дней × ссылки/тарифы/ОТО/бонусы/
миссии), `salebot_configs`, `product_durations`, дашборды, отчёты регистраций,
`bothelp_condition`, поиск/фильтры, экспорт, глубокая стыковка с UTM-нейминг-кодами.

## 5. Архитектура

```
ksamata-funnels-db/
  ksamata_funnels.db        # существующая БД (в проде — на volume)
  add_*.py                  # существующие утилиты (без изменений)
  app/                      # НОВОЕ: Next.js приложение
    src/
      db/                   # Drizzle: схема (отражает существующие таблицы) + клиент
      app/                  # роуты: / (список), /funnels/[id], /refs (справочники)
        api/                # REST-эндпоинты CRUD
      components/           # UI: FunnelCard, FunnelForm, StatusPill, CodeChip, RefTable
      lib/                  # дизайн-токены, хелперы AB-тегов
    Dockerfile
    drizzle/                # миграции (добавление status, front_code)
```

Единицы и их границы:
- **db/schema** — единственное место описания таблиц; всё типизировано из неё.
- **api/** — тонкий CRUD-слой: валидирует вход, пишет в БД через Drizzle, отдаёт JSON.
- **components/** — презентация; каждый компонент с одной задачей (карточка, форма,
  чип, таблица справочника).
- **lib/ab-tags** — единственное место логики «4 видимые АВ-оси ↔ строки `funnel_tags`»,
  чтобы UI не знал внутренностей тегов.

## 6. Поток данных

Браузер → Next API-роут → Drizzle → SQLite (volume). Чтение списка — server component.
Мутации (создать/править/статус/удалить) — через API-роуты с оптимистичным обновлением UI.
АВ-теги при сохранении воронки синхронизируются в `funnel_tags` (reg + time_19 + time_15)
по правилам `lib/ab-tags` (идемпотентно, как в `add_av_tags.py`).

## 7. Обработка ошибок

- Валидация на API: `num` уникален, `front_code` формата `f\d+`, обязательные FK
  существуют. Ошибка → 400 + понятное сообщение в UI (тост).
- Удаление справочного значения, если на него ссылается воронка → запрет с пояснением.
- БД-операции в транзакции; при ошибке — откат, UI не применяет оптимистичное изменение.
- Бэкап файла БД перед миграцией схемы (как `*.bak-before-av`).

## 8. Тестирование

- Юнит: `lib/ab-tags` (маппинг осей ↔ теги в обе стороны), валидаторы API.
- Интеграция: CRUD-эндпоинты против временной SQLite-копии (создать → прочитать →
  обновить статус → удалить; дубликат; добавление продукта).
- Ручная проверка: список рендерит 32 существующие воронки корректно; создание новой
  воронки (напр. f34 ТКМ) появляется в списке.

## 9. Первое наполнение (входит в работу)

Исходная задача становится сид-данными нового сервиса — **через UI или миграцию**:
- 3 новых продукта: **ТКМ, ЖИВО, СУСТАВЫ**.
- 6 новых воронок (скелет, num 33–38, front_code f32/f33/f34/f27/f29/f30):
  СУСТАВЫ·НИМБ·Яндекс·РСЯ, ЖИВО·НИМБ·Яндекс·РСЯ, ТКМ·НИМБ·Яндекс·РСЯ (черновик),
  ЖИВО·NR·ВК·Реклама, СВС·НИМБ·ВК·Реклама, ДЫХАНИЕ·FAQ·ВК·Реклама.
- Новые АВ-теги: `АВ Продукт: ТКМ/ЖИВО/СУСТАВЫ`, `АВ Подрядчик: FAQ`.
- Простановка `status` и `front_code` существующим 32 воронкам.

**Открытый вопрос (решить при наполнении):** чинить ли `num16` (вернуть
БОО/NR/ВК/In Stream, убрать мусорный тег «Яндекс Холодный квиз CHO») — вынесено отдельно,
не блокирует Фазу 1.

## 10. Критерии готовности Фазы 1

- Сервис собирается в Docker и деплоится через Dokploy; БД на volume.
- Список воронок выглядит как макет, отражает реальные данные из БД.
- Можно создать/отредактировать/продублировать/удалить воронку и сменить статус.
- Можно добавить новый продукт/подрядчик/источник/тег.
- Все 6 новых воронок и 3 продукта заведены; статусы и front_code проставлены.
