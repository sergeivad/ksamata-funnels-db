# Мониторинг доступности лендов

> Дата: 2026-07-24
> Статус: спека утверждена, план не написан

## Цель

Сервис сам регулярно проверяет доступность посадочных страниц из базы воронок и
показывает результат на отдельной странице `/monitoring` — дашборде со статусами.
Задача, которую это решает: упавший или молча уехавший на редирект ленд сейчас
обнаруживается только вручную и случайно, уже после потери трафика.

## Границы задачи (v1)

Входит:

- проверка доступности по HTTP, фоном, по расписанию;
- страница-дашборд с текущими статусами и историей инцидентов;
- возможность расширить охват с лендов на остальные ссылки базы переключателем.

Не входит (осознанно отложено):

- уведомления (Telegram, email) — сначала убеждаемся, что чеки адекватны;
- графики аптайма и SLA-проценты;
- проверка контента страницы (наличие ключевого текста, кнопки регистрации).

## Охват: что проверяем

Целей в базе потенциально ~830 URL, но проверять по умолчанию будем ~42 —
только ленды. Причина в шуме, а не в производительности:

| Источник | Кол-во | Почему |
|---|---|---|
| `lp.ksamata.ru`, `t.*.ru` (ленды) | ~42 | обычные страницы, 200 или упало — чистый сигнал |
| `gc.ksamata.ru` | 470 | GetCourse: часть страниц под авторизацией, вернёт 302/403 при живом сервисе |
| `salebot.pro`, `clck.ru` | ~70 | ссылки на ботов и редирект-сокращалки, HTTP-код мало что говорит |

Решение: в БД заводятся **все** URL как цели, но `enabled = 1` при первом
заведении получают только ленды. Остальные группы включаются галочкой на
странице. Расширение охвата — переключатель, а не переписывание.

Источники лендов — **два**, оба нужны:

- блок `funnel_blocks.kind = 'landings'` → 38 уникальных URL;
- `funnels.landing_url` → покрывает воронки №34, 36, 37, 38, у которых блок пуст.

## Модель данных (Phase 6)

Миграция `migrate-phase6.ts` + `migrate-phase6-data.ts` + `migrate-phase6-runner.ts`
по образцу Phase 5: идемпотентная (`CREATE TABLE IF NOT EXISTS`), добавляется
последним шагом в `app/docker-entrypoint.sh` после backfill-а тегов.

```sql
CREATE TABLE IF NOT EXISTS monitor_targets (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  url             TEXT    NOT NULL UNIQUE,
  source_kind     TEXT    NOT NULL,      -- 'landings' | 'funnel_landing_url' | 'links' | 'oto' | ... | 'manual'
  enabled         INTEGER NOT NULL DEFAULT 0,
  manual_override INTEGER NOT NULL DEFAULT 0,  -- 1, если enabled выставил человек тумблером
  note            TEXT    NOT NULL DEFAULT '',
  created_at      TEXT    NOT NULL DEFAULT (datetime('now')),
  updated_at      TEXT    NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS monitor_target_funnels (
  target_id INTEGER NOT NULL REFERENCES monitor_targets(id) ON DELETE CASCADE,
  funnel_id INTEGER NOT NULL REFERENCES funnels(id) ON DELETE CASCADE,
  PRIMARY KEY (target_id, funnel_id)
);

CREATE TABLE IF NOT EXISTS monitor_state (
  target_id            INTEGER PRIMARY KEY REFERENCES monitor_targets(id) ON DELETE CASCADE,
  status               TEXT    NOT NULL DEFAULT 'unknown'
                         CHECK(status IN ('up','slow','down','unknown')),
  http_status          INTEGER,
  final_url            TEXT    NOT NULL DEFAULT '',
  error                TEXT    NOT NULL DEFAULT '',
  latency_ms           INTEGER,
  checked_at           TEXT,
  since                TEXT,               -- когда статус стал текущим
  consecutive_failures INTEGER NOT NULL DEFAULT 0
);

CREATE TABLE IF NOT EXISTS monitor_events (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  target_id   INTEGER NOT NULL REFERENCES monitor_targets(id) ON DELETE CASCADE,
  from_status TEXT    NOT NULL,
  to_status   TEXT    NOT NULL,
  http_status INTEGER,
  error       TEXT    NOT NULL DEFAULT '',
  at          TEXT    NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX IF NOT EXISTS idx_monitor_events_target ON monitor_events(target_id);
CREATE INDEX IF NOT EXISTS idx_monitor_events_at     ON monitor_events(at);
```

Ключевое решение по объёму данных: **в `monitor_events` пишется только смена
статуса**, а не каждая проверка. Иначе при цикле раз в 15 минут и 830 целях база
растёт на ~80 тыс. строк в сутки. Текущее состояние живёт в `monitor_state`
(одна строка на цель, перезаписывается), история — десятки строк в месяц.

Связь «URL ↔ воронки» вынесена в отдельную таблицу, потому что один ленд
используется несколькими воронками (`lp.ksamata.ru/izh-yo` — в №2 и №4).
На дашборде это отвечает на вопрос «кого зацепит, если упало».

## Синхронизация целей

`app/src/lib/monitor-targets.ts`, функция `syncMonitorTargets()`:

1. Собирает URL из `funnel_block_items` (все виды блоков, `source_kind` = kind
   блока) и из `funnels.landing_url` (`source_kind = 'funnel_landing_url'`).
2. Нормализует: сплит многоссылочных значений по разделителю ` / `, `trim`,
   срезание мусорных хвостов (в №7 значение оканчивается кавычкой), отбрасывание
   всего, что не начинается с `http`. Схема (`http`/`https`) сохраняется как есть —
   `http://lp.ksamata.ru/izh-yo` проверяется именно по `http`.
3. Upsert по `url`. Один и тот же URL из разных источников — одна цель;
   `source_kind` берётся от первого источника по приоритету
   `landings` > `funnel_landing_url` > остальные.
4. Пересобирает связи в `monitor_target_funnels`.

Инварианты:

- **при первом заведении** `enabled = 1` только для `landings` и
  `funnel_landing_url`, для остальных `0`;
- у `enabled` два разных смысла — «человек так решил» и «вид источника такой», —
  и различает их колонка `manual_override`. Тумблер на странице (`setTargetEnabled`,
  `setSourceKindEnabled`) вместе с `enabled` ставит `manual_override = 1`;
- **при повторном upsert**: если `manual_override = 1`, `enabled` не трогается —
  ручное переключение переживает синк; если `manual_override = 0`, `enabled`
  пересчитывается из `source_kind`. Второе — обязательно: без него ленд, на один
  синк пропавший из данных воронок (и погашенный авто-ретайрментом), после
  возврата URL остался бы выключенным навсегда, да ещё и невидимо — под фильтром
  «показать выключенные»;
- цель, чей URL исчез из базы воронок, **не удаляется**: ей ставится
  `enabled = 0` и снимаются связи с воронками, история инцидентов сохраняется.
  Ретайрмент гасит цель независимо от `manual_override` — проверять URL, которого
  в данных больше нет, незачем. Когда URL возвращается, автоматически оживают
  только цели с `manual_override = 0`; цель, которую человек когда-то включил
  руками, придётся включить снова — зато это видно в таблице, а не молча;
- новой цели сразу создаётся строка `monitor_state` со статусом `unknown` —
  дашборд показывает «ещё не проверялось», а не пустоту.

Синк вызывается перед каждым прогоном цикла и кнопкой на странице.

## Чекер

`app/src/lib/monitor-check.ts` — чистая функция `checkUrl(url, opts)`, ничего не
знающая про БД: возвращает `{ status, httpStatus, finalUrl, latencyMs, error }`.
Такая граница позволяет тестировать её подменой `fetch`, без сети.

Одна проверка:

- `GET` через нативный `fetch`, `redirect: 'follow'`, `cache: 'no-store'`,
  заголовок `User-Agent: KsamataFunnelsMonitor/1.0`;
- таймаут 10 с через `AbortSignal.timeout(10_000)`;
- тело не читаем — поток обрывается после заголовков.

`GET`, а не `HEAD`, потому что часть лендов на кастомных доменах отвечает на
`HEAD` кодом 405, и мы получили бы ложные падения.

Классификация:

| Условие | Статус |
|---|---|
| финальный код 200, `latencyMs` ≤ 5000 | `up` |
| финальный код 200, `latencyMs` > 5000 | `slow` |
| любой другой код, таймаут, DNS/TLS-ошибка | `down` |

`finalUrl` пишется всегда. Отличие от исходного URL — не ошибка, но на дашборде
показывается стрелкой «→ куда увело»: молчаливый редирект ленда на чужую
страницу глазами не ловится, а это ровно тот случай, ради которого всё делается.

## Прогон цикла

`app/src/lib/monitor-run.ts`, функция `runMonitorCycle()`:

1. Вызывает `syncMonitorTargets()`.
2. Берёт цели с `enabled = 1`, гоняет их пулом по 8 параллельно.
3. Для каждой цели перезаписывает `monitor_state`.
4. Пишет строку в `monitor_events` **только если статус изменился**; тогда же
   обновляет `since`.

Защита от ложных инцидентов: падение подтверждается **внутри одного цикла** —
провалилась проверка, пауза 3 секунды, повторная проверка; статус `down`
ставится, только если провалились обе попытки. Реальное падение попадает в
`down` за один цикл, а не за два. `consecutive_failures` считает подряд идущие
циклы с подтверждённым падением и нужен для отображения («падает N циклов
подряд»); `slow` — рабочее состояние, счётчик не трогает; любой не-`down`
исход счётчик обнуляет.

Модульный флаг «цикл уже идёт» не даёт наложиться планировщику и ручной кнопке.

## Планировщик

`app/instrumentation.ts` (штатный хук Next 15): при старте на рантайме `nodejs`
заводит `setInterval`.

| Переменная | По умолчанию | Смысл |
|---|---|---|
| `MONITOR_ENABLED` | `true` | `false` полностью выключает фон (локально, чтобы dev-сервер не долбил боевые ленды) |
| `MONITOR_INTERVAL_MINUTES` | `15` | период цикла |

Первый прогон — через 30 секунд после старта, чтобы не конкурировать с
миграциями из entrypoint. Инстанс один, поэтому дублей нет; при появлении
второго спасёт in-flight флаг плюс проверка свежести `checked_at`.

## API

`app/src/app/api/monitoring/`, по образцу существующих роутов — Zod-валидация,
хелперы `http.ts` / `errors.ts`.

| Метод и путь | Назначение |
|---|---|
| `GET /api/monitoring` | сводка + цели с состоянием и связанными воронками |
| `POST /api/monitoring/run` | запускает синк + прогон и отвечает `202` сразу, не дожидаясь конца (см. ниже); `409`, если цикл уже идёт |
| `PATCH /api/monitoring/targets/[id]` | `{ enabled }` для одной цели |
| `PATCH /api/monitoring/targets` | групповое `{ sourceKind, enabled }` |
| `GET /api/monitoring/events` | история инцидентов, с пагинацией |

Прогон не дожидается ответа сознательно. На охвате по умолчанию цикл занимает
секунды, но группа GetCourse (470 целей) включается одним кликом, а одна
недоступная цель стоит до 23 с (таймаут 10 с + пауза 3 с + ретрай 10 с). При
параллельности 8 это десятки минут: обратный прокси оборвал бы соединение,
страница показала бы «Не удалось запустить проверку» при живом цикле, и оператор
нажал бы кнопку ещё раз. Поэтому роут отдаёт `202`, а страница опрашивает
`GET /api/monitoring` раз в 2 секунды, пока `summary.running` не станет `false`,
и по дороге обновляет таблицу.

## Страница `/monitoring`

Пункт «Мониторинг» в `AppHeader` рядом с «Тегами».

- **Полоса сводки:** всего / работает / медленно / упало, «последняя проверка N
  минут назад», кнопка «Проверить сейчас». Кнопка в состоянии «Проверяем…»,
  пока `summary.running` истинно, — в том числе для цикла, запущенного
  планировщиком; на `409` — сообщение «проверка уже идёт».
- **Фильтры:** статус, поиск по URL, переключатель «показать выключенные (N)».
  По умолчанию выключенные цели скрыты — иначе ~790 серых строк похоронят 42
  полезные.
- **Таблица**, строка на URL: пилюля статуса (`StatusPill`), URL внешней
  ссылкой, под ним `→ финальный URL` при редиректе, HTTP-код, время ответа,
  «в этом статусе с …», чипы с номерами воронок (ведут на карточку),
  тумблер вкл/выкл. Сортировка: красные наверх, затем жёлтые.
- **Расширение охвата:** строка чипов над таблицей — `links · 164 · выкл`,
  `oto · 62 · выкл` и т. д., клик включает группу целиком.
- **«Последние события»:** сжатый лог смен статуса вида
  `№12 t.spina-pozvon.ru: работает → упало (503), 3 часа назад`.

Компоненты — `app/src/components/monitoring/`: `MonitorSummary`,
`MonitorTable`, `MonitorEvents`. Страница остаётся тонкой, ни один файл не
разрастается.

## Тесты

`vitest`, по образцу существующих — на временной копии `ksamata_funnels.db`.

- `monitor-check.test.ts` — подменённый `fetch`: 200 → `up`; медленный 200 →
  `slow`; 404 и 500 → `down`; таймаут → `down` с текстом ошибки; редирект →
  `finalUrl` записан; запрос уходит методом `GET`.
- `monitor-targets.test.ts` — синк: сплит многоссылочного `landing_url` (кейс
  №7 с хвостовой кавычкой); дедупликация одного URL из двух воронок с обеими
  связями; `enabled = 1` только у лендов; повторный синк не сбрасывает ручной
  тумблер; исчезнувший URL гаснет, но не удаляется; пропавший и вернувшийся ленд
  включается обратно сам (`manual_override = 0`).
- `monitor-run.test.ts` — `monitor_events` пишется только при смене статуса;
  одиночная неудача не роняет в `down`, вторая подряд — роняет; возврат в `up`
  закрывает инцидент и обновляет `since`.
- `migrate-phase6.test.ts` — идемпотентность (двойной прогон миграции).
- `api-monitoring-route.test.ts` — контракты роутов, включая `409` на
  параллельный прогон.

Проверка на выходе: `npx tsc --noEmit`, `npx vitest run`, `npm run build`.
