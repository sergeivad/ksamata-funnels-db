# Дизайн: гибкое управление АВ-тегами

Дата: 2026-07-19
Статус: утверждён, готов к плану реализации

## Проблема

Блок «АВ-теги · сценарий предложения» на карточке воронки сейчас **не редактируется**.
Теги вычисляются на лету из 4 осей (продукт / подрядчик / канал / направление)
функцией `axesToTagNames()` и служат только для копирования в буфер. При каждом
сохранении идентификации `syncAvTags()` **стирает все `АВ …` теги и пересобирает
их заново** из осей. Нельзя добавить свой тег и нельзя убрать дефолтный.

Нужно: добавлять свои теги, удалять дефолтные — гибко, на уровне воронки **и**
глобально (шаблон для всех).

## Утверждённые решения

1. **Уровень:** и глобальный шаблон, и переопределение на воронке.
2. **Поведение при смене оси:** Вариант А — ручные правки живут (добавленные теги
   остаются, вычеркнутые не возвращаются); обновляется только то, что зависит от осей.
3. **Гранулярность на воронке:** по сценарию (Регистрация / Оплата-15 / Оплата-19 /
   Мессенджер — отдельные наборы правок).
4. **Что редактируется в глобальном шаблоне:** только статичные теги
   (`автоворонки`, `АВ Автоворонка`, `АВ Этап: …`, `АВ Время: …`). Префиксы
   axis-тегов (`АВ Продукт: ` и т.д.) остаются в коде, axis-теги — автогенерация.
5. **Вычеркнутый дефолт** показывается бледным/зачёркнутым с кликом-восстановлением
   (всегда видна полная картина), а не просто исчезает.
6. **Отдельная кнопка «Сохранить теги»** в блоке (не объединять с «Сохранить
   идентификацию»).
7. **Правка глобального шаблона сразу пересобирает все воронки** (бэкфилл), не лениво.

## Ключевая идея — «слои»

Итоговый набор тегов для сценария `S` собирается из слоёв:

```
Итог(S) =
    ШАБЛОН.статичные[S]        ← глобально редактируемые дефолты
  + АВ-теги осей(воронка)       ← авто из осей (как сейчас)
  − removed[воронка, S]         ← вычеркнутые дефолты (по имени)
  + added[воронка, S]           ← добавленные вручную теги
```

Порядок в итоге: `шаблон → оси → added`, с дедупом по точному совпадению строки.

**Почему Вариант А получается сам собой:** оси не хранятся как явные теги в
overrides — они всегда берутся из живого слоя «АВ-теги осей». Пользователь может
только *добавить своё* и *вычеркнуть дефолт по имени*. Поэтому смена продукта
СУСТАВЫ→ЖКТ обновляет `АВ Продукт:` в слое осей, а `added`/`removed` остаются
нетронутыми — никакой заморозки axis-тегов.

## Критический факт архитектуры

`channel` и `direction` **не имеют** FK-колонок на `funnels` — они хранятся
**только** как теги `АВ Канал:` / `АВ Направление:` в `funnel_tags`. Значит
`funnel_tags` одновременно и материализованный вывод, и источник истины для двух
осей. Поэтому **во всех путях материализации сначала читаем `axes`
(из формы или `getAxesForFunnel`), и только потом переписываем `funnel_tags`.**

## Данные — 2 новые таблицы

### `tag_templates` — глобальный шаблон статичных тегов

| поле | тип | |
|---|---|---|
| id | integer PK autoincrement | |
| scenario | text | `reg` \| `time_15` \| `time_19` \| `messenger` |
| name | text | текст тега |
| position | integer | порядок внутри сценария |

Сидируется текущими зашитыми значениями:
- `reg`: `автоворонки`, `АВ Автоворонка`, `АВ Этап: Регистрация`
- `time_19`: `автоворонки`, `АВ Автоворонка`, `АВ Этап: Оплата`, `АВ Время: 19`
- `time_15`: `автоворонки`, `АВ Автоворонка`, `АВ Этап: Оплата`, `АВ Время: 15`
- `messenger`: `автоворонки`, `АВ Автоворонка`, `АВ Этап: Мессенджер`

### `funnel_tag_overrides` — правки на воронке

| поле | тип | |
|---|---|---|
| id | integer PK autoincrement | |
| funnel_id | integer FK → funnels (cascade) | |
| tag_type | text | `reg` \| `time_15` \| `time_19` \| `messenger` |
| name | text | текст тега |
| op | text | `add` \| `remove` |
| position | integer | порядок для `add` |
| unique | (funnel_id, tag_type, name) | |

Хранение **по имени**, не по `tag_id`: `remove автоворонки` остаётся стабильным
при любой churn в таблице `tags`; `add` создаёт `tags`-строку лениво при
материализации.

### `funnel_tags` — без изменений схемы

Остаётся материализованным итогом (его читают экспорт, `getAxesForFunnel`,
дубликаты). Меняется только способ сборки.

## Логика

### `ab-tags.ts`
- Убрать зашитые `COMMON_TAGS` / этапы / время (переезжают в сид `tag_templates`).
- Оставить `AXIS_PREFIXES`, `AbAxes`, `tagNamesToAxes`.
- Добавить `axisTagNames(axes): string[]` — только axis-теги (`АВ Продукт:` и т.д.),
  с фильтром пустых осей (как сейчас).
- Добавить чистую `computeTagSet(template, axes, overrides) → {reg, time_15, time_19,
  messenger}` — дедуп + порядок `шаблон → оси → added`, минус `removed`.

### `funnels.ts`
- `syncAvTags` → **`materializeFunnelTags(tx, funnelId, axes)`**: грузит шаблон +
  overrides, считает `computeTagSet`, **полностью переписывает** `funnel_tags`
  воронки. Спецслучай «сохранить legacy non-AV теги» уходит — итог самодостаточен.
- Во всех вызовах: сначала читать `axes`, потом переписывать `funnel_tags`.
- `duplicateFunnel` / `copyFunnelChildren`: копировать `funnel_tag_overrides`
  (а не «legacy non-AV tags»).
- Новая `resyncAllFunnels(db)` (бэкфилл): пересобрать `funnel_tags` у всех воронок,
  сохраняя overrides. Вызывается после правки глобального шаблона.

## API

### Глобальный шаблон (по образцу `/api/refs`)
- `GET /api/tag-templates` → `{reg:[…], time_15:[…], time_19:[…], messenger:[…]}`
- `PUT /api/tag-templates/[scenario]` → замена всего упорядоченного списка сценария
  (`{names: string[]}`), затем `resyncAllFunnels`.

### Overrides воронки
- `getFunnel` дополняется полем `tagSets`: по сценарию массив
  `{name, source: 'default'|'custom'}` (итог с пометкой происхождения — для × и стилей).
- `PATCH /api/funnels/[id]/tags` → тело `{reg:{add:[],remove:[]}, time_15:{…}, …}`;
  сервер заменяет `funnel_tag_overrides` воронки и материализует. Клиент строит
  `add/remove` сам, зная `source` каждого чипа.

### Валидация (`validation.ts`, zod)
- Имя тега: trimmed, `min(1)`, `max(120)` (как `REF_MAX`).
- `scenario` / `tag_type`: enum из 4 значений.
- Тело `PATCH …/tags`: объект по сценариям с массивами строк `add`/`remove`.

## UI

### Блок «АВ-теги» в `FunnelIdentity` — редактируемый
- На активной вкладке сценария у каждого чипа — маленький `×`.
- Поле «+ добавить тег» (Enter добавляет).
- Кастомные чипы визуально отличаются (др. фон / точка).
- Вычеркнутый дефолт — бледный/зачёркнутый, клик восстанавливает.
- Свой dirty-индикатор + кнопка «Сохранить теги» (отдельно от «Сохранить
  идентификацию»). Подсказка «сначала сохраните идентификацию», если оси не сохранены
  (набор дефолтов зависит от сохранённых осей).
- Клиент рендерит `tagSets` с сервера (больше не считает `axesToTagNames` локально);
  из пометок `source` строит `add/remove` для PATCH.

### Новый экран `/tags` (по образцу `/refs` + `RefTable`)
- 4 секции сценариев; в каждой — упорядоченный список чипов
  (добавить / удалить / reorder) + сохранение.
- Ссылка из `AppHeader`.
- Axis-теги показаны read-only подсказкой «добавляются автоматически».

## Миграция и сид
- Phase-5 скрипт (DDL двух таблиц + сид `tag_templates` из текущих констант) по
  образцу `migrate-phaseN.ts` + runner.
- Обновить baked seed DB в `app/seed/` (как коммит «refresh baked seed DB»).

## Тесты
- `computeTagSet`: дедуп, порядок, пустые оси, removed/added.
- `materializeFunnelTags` с overrides.
- **Вариант А:** смена оси сохраняет `add`/`remove`.
- Бэкфилл от правки шаблона сохраняет overrides у воронок.
- CRUD шаблона (`GET`/`PUT`) и overrides (`PATCH`).
- `duplicateFunnel` копирует overrides.

## Вне области (YAGNI)
- Редактирование префиксов axis-тегов (`АВ Продукт: ` и т.д.).
- Переопределение шаблона на группу воронок (только глобально + per-funnel).
- Версионирование/история изменений тегов.
