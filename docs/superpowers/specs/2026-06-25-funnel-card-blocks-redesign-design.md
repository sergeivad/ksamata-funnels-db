# Funnels Admin — Карточка воронки: блоки ссылок — дизайн (черновик)

**Дата:** 2026-06-25 (обновлено 2026-06-26 — решения по открытым вопросам)
**Статус:** ЧЕРНОВИК (шаблон v8 одобрен; решения по данным/миграции — §10)
**Репозиторий:** `ksamata-funnels-db`, приложение в `app/` (продолжение Фаз 1–2)
**Утверждённый макет:** `funnel_card_with_identity_v8` (визуальный черновик в сессии)
**Референс структуры/нумерации:** `ksamata-leak-funnels/F*/F*.png` — скриншоты
наборов правил атрибуции (`leak.besales.ai`, экран «Воронки / Правила») по
каждому `f#`: «Коды вебинарных комнат» (напр. F12 ЖКТ `zkt1-19-nr…`), «Макс.
день вебинара» и т.д. Используются как эталон при сверке миграции.

## 1. Зачем

Сейчас карточка воронки показывает идентичность отдельной формой сверху и
сетку «5 дней × 2 времени» (`DaysEditor`) только с тремя полями (GC-комната,
Web-комната, ленд). При этом в `funnel_days` уже лежит 17 колонок и 300
заполненных строк (тарифы, ОТО, бонусы, процессы, повторы, медитации) —
невидимых в UI. Источник данных — `Воронки ссылки-5.xlsx` (лист на продукт,
9 типов ссылок по дням × времени).

Цель: пересобрать карточку в **одну цельную форму** — идентификация сверху,
ниже — тематические блоки ссылок, — и сделать доступными для редактирования
все типы ссылок, разложенные по понятным блокам.

## 2. Структура карточки (сверху вниз)

### 2.1. Идентификация
- **Код `f#`** — редактируемое поле-плашка слева сверху (`funnels.front_code`).
- **Имя воронки** — заголовок, **только для чтения**, собирается из осей:
  `«{Продукт} / {Подрядчик} / {Канал} / {Направление}»`
  (напр. `БОО / NR / ВК / Перелив с БОО`). Отдельного поля «Вариант/название» нет.
- **Статус** — сегмент Активна / Черновик (`funnels.status`).
- **Оси (селекты с добавлением нового):** Продукт, Подрядчик, Канал, Направление.
  Источники опций — `products`, `contractors`, `channels`, `directions`. АВ-теги
  генерируются из осей при сохранении.
- **Комментарий** — свободный многострочный текст (новая колонка
  `funnels.comment`).
- **АВ-теги** — read-only плашка-список: `Продукт: X`, `Подрядчик: X`,
  `Канал: X`, `Направление: X`, `Автоворонка`, `Этап: Регистрация/Оплата`,
  `Время: A / B`. Генерируются автоматически (`axesToTagNames`), руками не правятся.
- **Убрано из шапки:** «Вариант/название», «Дата старта», «Лендинг» (переехал в
  блок), «Блок» (убрано в Фазе 2). Колонки `funnels.variant`, `start_date`,
  `landing_url` остаются в БД, но не редактируются (`landing_url` мигрирует в
  блок «Лендинги», см. §5).

### 2.2. Время воронки
- Два редактируемых поля, по умолчанию **15:00** и **19:00**; значения можно
  изменить на любые. Хранятся в новых колонках `funnels.time_label_a` (деф.
  `15:00`) и `time_label_b` (деф. `19:00`).
- **Внутренние ключи слотов неизменны:** `A = '15'`, `B = '19'` (идентификаторы,
  не отображаемый текст). Используются в `funnel_days` и `funnel_block_items.slot`.
  Меняется только подпись (`time_label_*`) — так не трогаем
  `CHECK(time_slot IN ('19','15'))` и существующие строки.
- Порядок везде: сначала слот A (15), затем слот B (19).

### 2.3. Блоки — порядок «по пути клиента»
1. **Лендинги** — список, одно поле, всегда «Общее» (вход).
2. **Вебинарные комнаты** — особый блок (см. §3).
3. **Записи** — список, одно поле (запись/повтор вебинара).
4. **Страницы тарифов** — список, одно поле.
5. **Оформление заявки** — список, одно поле.
6. **Бонусы** — список, одно поле.
7. **ОТО** — список, одно поле.
8. **Процессы** — список, два поля (описание + ссылка).
9. **Медитация / дожим** — список, одно поле.
10. **Ссылки / дашборды** — список, два поля (служебное, низ).

## 3. Блок «Вебинарные комнаты» (особый)

- Всегда активен (нельзя выключить), всегда показывает **оба времени**
  колонками: слот A (15:00) и слот B (19:00), рядом.
- Каждый слот — строки по дням. Поля на ячейку: **GC-комната**, **Web-комната**
  (повтора здесь больше нет — он в блоке «Записи»).
- **Число дней — динамическое:** по умолчанию **3** строки (дня), кнопка
  «+ добавить день» добавляет следующий день (как в блоках-списках). Текущий
  предел БД — 5 дней (`CHECK(day_num BETWEEN 1 AND 5)`); если понадобится
  больше — отдельная правка (rebuild `funnel_days`). Для существующих воронок
  число дней = макс. `day_num` с данными (для новых — 3 пустые).
- Хранение: остаётся таблица `funnel_days`, используется подмножество колонок:
  `gc_room`, `web_room`. Прочие колонки после миграции (§5) не редактируются.
- Плейсхолдеры в пустых полях.

## 4. Блоки-списки (единый компонент)

Все блоки кроме Комнат — один переиспользуемый компонент, параметризованный
**видом блока** (`kind`). Поведение:

- **Тумблер вкл/выкл** — активация блока для воронки (`funnel_blocks.enabled`).
  Выключенный блок свёрнут и приглушён.
- **Режим** (`funnel_blocks.mode`): `common` («Общее») или `by_time`
  («По времени»). Сегмент-переключатель в шапке.
  - `common` → один список строк.
  - `by_time` → две подколонки, подписанные `time_label_a` / `time_label_b`,
    каждая со своим списком строк.
  - Исключение: **Лендинги** — всегда `common`, переключатель режима скрыт.
- **Поля строки** — свойство вида блока (в коде, не в БД):
  - **одно поле** (только ссылка): Лендинги, Записи, Страницы тарифов,
    Оформление заявки, Бонусы, ОТО, Медитация/дожим.
  - **два поля** (описание + ссылка): Процессы, Ссылки/дашборды.
- Действия: добавить строку, удалить строку, переупорядочить (`position`).
- Ссылка рендерится кликабельной только при `http(s)://` (гард `isSafeUrl`).
- Плейсхолдеры во всех пустых полях; цвет плейсхолдера `#8A8275`.

**Каталог видов блоков** (константа `app/src/lib/blocks.ts`, порядок = §2.3):

| kind | Заголовок | Иконка | Полей | Режимы | Деф. для НОВОЙ воронки |
|------|-----------|--------|-------|--------|------------------------|
| `landings`     | Лендинги           | browser             | 1 | common          | вкл |
| `records`      | Записи             | video               | 1 | common, by_time | выкл |
| `tariffs`      | Страницы тарифов   | discount            | 1 | common, by_time | вкл |
| `applications` | Оформление заявки  | file-invoice        | 1 | common, by_time | вкл |
| `bonuses`      | Бонусы             | gift                | 1 | common, by_time | выкл |
| `oto`          | ОТО                | flame               | 1 | common, by_time | выкл |
| `processes`    | Процессы           | settings-automation | 2 | common, by_time | выкл |
| `meditation`   | Медитация / дожим  | yoga                | 1 | common, by_time | выкл |
| `links`        | Ссылки / дашборды  | link                | 2 | common, by_time | вкл |

(Колонка «деф.» применяется только при создании новой воронки; для существующих —
определяется миграцией §5.2: блок включается, если для него нашлись данные.)

## 5. Схема БД и миграция

### 5.1. Изменения схемы (идемпотентная миграция)

```sql
-- Идентификация / время
ALTER TABLE funnels ADD COLUMN comment      TEXT DEFAULT '';
ALTER TABLE funnels ADD COLUMN time_label_a TEXT DEFAULT '15:00';
ALTER TABLE funnels ADD COLUMN time_label_b TEXT DEFAULT '19:00';

-- Конфиг блоков на воронку
CREATE TABLE IF NOT EXISTS funnel_blocks (
  id        INTEGER PRIMARY KEY AUTOINCREMENT,
  funnel_id INTEGER NOT NULL REFERENCES funnels(id) ON DELETE CASCADE,
  kind      TEXT    NOT NULL,
  enabled   INTEGER NOT NULL DEFAULT 0,
  mode      TEXT    NOT NULL DEFAULT 'common' CHECK(mode IN ('common','by_time')),
  UNIQUE(funnel_id, kind)
);

-- Строки блоков
CREATE TABLE IF NOT EXISTS funnel_block_items (
  id       INTEGER PRIMARY KEY AUTOINCREMENT,
  block_id INTEGER NOT NULL REFERENCES funnel_blocks(id) ON DELETE CASCADE,
  slot     TEXT    DEFAULT NULL CHECK(slot IN ('15','19') OR slot IS NULL),
  label    TEXT    NOT NULL DEFAULT '',
  url      TEXT    NOT NULL DEFAULT '',
  position INTEGER NOT NULL DEFAULT 0
);
CREATE INDEX IF NOT EXISTS idx_fbi_block ON funnel_block_items(block_id);
```

- `slot` = NULL для строк в режиме `common`; `'15'`/`'19'` для `by_time`.
- Таблица `funnel_links` (0 строк) поглощается моделью как `kind='links'` и далее
  не используется.
- `funnel_days` без изменений схемы; UI редактирует только `gc_room`/`web_room`.

### 5.2. Перенос существующих данных (по каждой воронке)

Источник — `funnel_days` (300 строк) + колонки `funnels`; сверка структуры/кодов
комнат — по скриншотам правил `ksamata-leak-funnels/F*/F*.png`. Для каждой воронки
создаём `funnel_blocks` (enabled=1, если найдены непустые данные) и наполняем
`funnel_block_items`:

| Источник | → Блок (`kind`) | slot | label | url |
|----------|------------------|------|-------|-----|
| `funnels.landing_url` | `landings` | NULL (common) | '' | landing_url |
| `funnel_days.replay_url`, `web_replay` | `records` | time_slot | '' | url |
| `funnel_days.sales_page` | `applications` | time_slot | '' | sales_page |
| `funnel_days.tariffs` | `tariffs` | time_slot | '' | tariffs |
| `funnel_days.oto` | `oto` | time_slot | '' | oto |
| `funnel_days.bonuses` | `bonuses` | time_slot | '' | bonuses |
| `funnel_days.mission` | `processes` | time_slot | `mission_type` (или `«День N»`) | mission |
| `funnel_days.meditation` | `meditation` | time_slot | '' | meditation |
| `funnels.dash_sales_url`, `dash_pereliv_url`, `regi_total_url`, `regi_15_url`, `regi_19_url`, `regi_notime_url`, `predspisok_url` (непустые) | `links` | NULL | имя-метка колонки | URL |

- **НЕ переносим** `funnel_days.sales_note` и `dojim_note` (замусоренные
  заметки — по решению пользователя).
- **Комнаты:** `gc_room`/`web_room` остаются в `funnel_days`; повтора в комнатах
  больше нет (переехал в `records`).
- **Режим блока после миграции:** если непустые значения только в одном слоте →
  `common` (slot строк → NULL); если в обоих → `by_time`.
- Миграция **идемпотентна** и **не трогает** уже заполненные вручную
  `funnel_blocks`/`funnel_block_items` (повторный запуск — no-op при наличии
  блоков у воронки).
- **Бэкап БД перед миграцией** (как в Фазе 2).

## 6. API

- `GET /api/funnels/[id]` → идентичность + производное имя + АВ-теги +
  `time_label_a/b` + блоки с их items.
- `PUT /api/funnels/[id]` → идентичность: `frontCode`, `status`, оси
  (product/contractor/channel/direction), `comment`, `time_label_a/b`.
  Авто-источник и АВ-теги пересобираются как в Фазе 2.
- `GET/PUT /api/funnels/[id]/rooms` → сетка 2 слота × N дней (`gc_room`,
  `web_room`); reconcile поверх `funnel_days` (UPDATE только этих колонок,
  прочие сохраняются); поддержка добавления/удаления дня.
- `GET/PUT /api/funnels/[id]/blocks/[kind]` → `{ enabled, mode, items }`;
  PUT в одной транзакции: upsert конфига + replace строк (delete-all + insert,
  `position` = индекс, `slot` по режиму).
- `/api/refs/[kind]` — без изменений.

## 7. Компоненты (UI)

- `FunnelIdentity` — шапка: редактируемый код-плашка, производное имя (read-only),
  статус-сегмент, селекты осей (RefSelect с добавлением), комментарий,
  read-only АВ-теги.
- `TimeLabels` — два редактируемых поля времени.
- `RoomsEditor` — 2 слота × N дней (GC/Web), кнопка «добавить день», деф. 3.
- `BlockEditor` — generic по `kind`: шапка (тумблер вкл/выкл, сегмент режима
  если режимов >1), список строк (1 или 2 поля), добавить/удалить/сортировать;
  `common` → один список, `by_time` → две подколонки по `time_label_*`.
- `app/src/app/funnels/[id]/page.tsx` — компонует: `FunnelIdentity` →
  `TimeLabels` → блоки в порядке §2.3 (`RoomsEditor` среди них по позиции).
- Список воронок (`app/src/app/page.tsx`) — имя в формате
  `«Продукт / Подрядчик / Канал / Направление»`.

## 8. Дизайн / стиль

Кремовый «ксаматовский» стиль (`app/src/lib/tokens.css`): тёплый фон, `#111`
акценты, оранжевый `#FF6A13` для активных тумблеров/действий, моно-чипы кодов и
ссылок, мягкие карточки. Тёмные плейсхолдеры (`#8A8275`). Без поясняющих
подписей-«лесов» в живом UI.

## 9. Что НЕ меняем

АВ-логика/таксономия тегов; существующие колонки `funnel_days` (после миграции
устаревшие, не удаляются); деплой (Docker/Dokploy) — новые таблицы создаются
идемпотентной миграцией, в т.ч. при первом старте на volume, как в Фазе 2.

2. **Спорные заметки** — ✅ `sales_note` и `dojim_note` НЕ переносим. Добавлен

4. **Дашборд-колонки** — ✅ переносим в блок «Ссылки/дашборды».

## 11. Тестирование

- Юнит (temp-копия БД): миграция схемы + перенос данных (раскладка по блокам,
  определение режима common/by_time, перенос повторов в `records`, дашбордов в
  `links`, пропуск sales_note/dojim_note); `funnel-blocks` reconcile (upsert
  конфига + replace строк, slot по режиму); производное имя из осей; АВ-теги из
  осей; rooms reconcile (добавление/удаление дня, сохранность прочих колонок).
- Сборка и полный прогон Vitest зелёные; ручная проверка карточки: шапка,
  комнаты (3 дня + добавить), блок в `common` и `by_time`, вкл/выкл блока, `/refs`.

## 12. Критерии готовности

- Карточка = одна форма: идентичность (код в плашке, имя из осей, оси-селекты,
  комментарий, авто-теги) + время (деф. 15/19, редактируемое) + блоки.
- Комнаты = GC + Web, оба времени (15→19), дни динамические (деф. 3 + добавить).
- Блоки-списки: вкл/выкл, режим Общее/По времени (Лендинги — всегда Общее),
  одно/два поля по каталогу; добавление/удаление/сортировка; плейсхолдеры.
- Существующие данные перенесены в новую модель миграцией; деплой не сломан.
- Тесты и сборка зелёные.
